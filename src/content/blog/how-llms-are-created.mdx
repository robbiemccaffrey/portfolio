## How a Language Model Gets Built

Most people now interact with ChatGPT, Claude, or Gemini daily — asking questions, writing code, summarising documents. But very few understand what actually goes into building one. The gap between "I use it" and "I understand it" is enormous, and closing that gap changes how you think about what these systems can and can't do.

The single best resource I've found for understanding this is Andrej Karpathy's [Deep Dive into LLMs like ChatGPT](https://www.youtube.com/watch?v=7xTGNNLPyMI). This post distils the key ideas from that talk and extends them with what changed in 2025 — specifically, the emergence of RLVR as a fourth training stage that unlocked genuine reasoning capabilities.

---

## Stage 1: Pretraining — Reading the Internet

Everything starts with data. Pretraining is the foundational stage where a model ingests an enormous corpus of text — web pages, books, Wikipedia, code repositories, forums, academic papers — and learns statistical patterns across all of it.

The scale is staggering. Datasets like [FineWeb](https://huggingface.co/datasets/HuggingFaceFW/fineweb) contain over 1.2 billion web pages, heavily filtered for quality. Raw web crawls are noisy — full of spam, duplicates, and low-quality content — so significant effort goes into cleaning and deduplication before a single token is processed.

### Data Curation — The Hidden Lever

It's worth noting how much smarter data curation has become. When GPT-2 was trained in 2019, the approach to data was relatively crude — scrape a lot of the web and do basic filtering. ChatGPT launched in late 2022 on a much larger but still imperfect dataset. Since then, the industry has learned that **data quality matters as much as data quantity**. Modern pretraining pipelines use aggressive deduplication (near-duplicate detection at scale), quality classifiers trained to score every page, toxicity and PII filters, domain-specific heuristics (is this machine-generated spam?), and careful balancing across domains — more code, more scientific text, less SEO content. FineWeb, for example, starts from 96 billion web pages and filters down to roughly 15 billion high-quality pages. The same model architecture trained on better-curated data produces a meaningfully smarter base model. Data curation has become one of the most important — and least visible — competitive advantages in AI.

This raises an obvious question: if quality matters so much, why not just train on books? Books are well-edited, factually reviewed, grammatically correct, and cover every subject in depth. Surely a model trained on nothing but books would be better than one trained on the messy internet?

It wouldn't — and the reason is **diversity**. A model trained only on books would write beautifully but fail at almost everything else people actually need. It wouldn't understand code (books rarely contain Python). It wouldn't understand conversational language (books don't have Reddit threads, Slack messages, or StackOverflow answers). It wouldn't know how to parse a JSON object, write a SQL query, or interpret a medical lab result. It would have no exposure to the way real people actually communicate — slang, abbreviations, typos, multilingual mixing.

The data mixture is itself a carefully tuned hyperparameter. Modern training runs typically use something like 50% web text, 25% code, 15% books and academic papers, and 10% conversational and reference data — though the exact ratios are closely guarded secrets. The insight is that breadth of capability comes from breadth of data. Quality filtering removes the garbage; diversity ensures the model can actually do what people ask it to do.

### Neural Networks — A Quick Primer

Before diving into tokenization and transformers, it's worth briefly covering what a neural network actually is — since everything that follows is built on top of one.

A neural network is a mathematical function loosely inspired by the brain. It's made of layers of **neurons** — each neuron takes some numbers as input, multiplies them by **weights** (learnable parameters), adds them up, and passes the result through an activation function. Stack millions of these neurons in layers, and you get a system that can learn complex patterns from data.

### Parameters — What "175 Billion Parameters" Actually Means

You'll hear models described by their parameter count — GPT-3 has 175 billion parameters, Llama 3 has 70 billion, GPT-4 has an estimated 1.8 trillion. But what *is* a parameter?

A parameter is a single number — a weight or a bias — that the model adjusts during training. Each connection between neurons has a weight (how strongly one neuron influences another), and each neuron has a bias (a baseline offset). Together, these numbers define everything the model "knows." The entire intelligence of the system is encoded in these numbers — nothing else.

Every one of these parameters starts as a random number. Training is the process of adjusting each one — billions of tiny nudges, repeated trillions of times — until the model can predict the next token accurately. When people say a model "learned" something, what they mean is: the right combination of parameter values was found to make that behaviour emerge from next-token prediction.

### The Loss Function — How the Network Learns

Training works by showing the network examples, measuring how wrong its predictions are, and nudging all the weights slightly in the direction that reduces the error. The measurement of "how wrong" is called the **loss function**. For language models, the loss is simple: the model predicts a probability distribution over the next token, and the loss measures how far that prediction was from the actual next token. If the model assigns 90% probability to the correct token, the loss is low. If it assigns 2%, the loss is high.

An algorithm called **backpropagation** then traces backwards through the network, calculating how much each individual weight contributed to the error. Every weight gets a tiny nudge in the direction that would have made the prediction better. Multiply this by billions of weights and trillions of training examples, and the weights gradually converge on values that make the network remarkably good at its task. The key insight: nobody programs the rules. The network discovers them from the data.

### Why GPUs Are So Expensive (and in Such High Demand)

Training a model like GPT-4 requires thousands of GPUs running continuously for months. But why GPUs? The answer is parallelism. A CPU handles tasks sequentially — it's fast at doing one thing at a time. A GPU was originally designed for rendering millions of pixels simultaneously on a screen. It turns out that the mathematical operations behind neural networks — multiplying huge matrices of numbers — are structurally identical to the operations behind rendering graphics. Thousands of simple cores doing the same operation in parallel is exactly what training needs.

NVIDIA's H100 GPU — the workhorse of modern AI training — costs roughly $30,000-$40,000 per unit. A serious training cluster needs thousands of them. Meta's Llama 3 training used 16,384 H100s running for weeks. At cloud rates, a single H100 rents for roughly $2-3 per hour. A 10,000-GPU cluster at $2.50/hour burns **$600,000 per day** — just on compute, before electricity, cooling, networking, and engineering salaries.

A **transformer** is a specific type of neural network architecture, introduced in the 2017 paper ["Attention Is All You Need"](https://arxiv.org/abs/1706.03762). What makes transformers special is the **attention mechanism** — the ability to look at all previous tokens simultaneously and decide which ones are relevant for predicting the next one. Previous architectures (RNNs, LSTMs) processed tokens sequentially, like reading one word at a time and trying to remember. Transformers can look at the whole sentence at once. This is what made them scale — and it's why every modern LLM is built on them.

Before any of this learning can happen, text needs to be converted into something a neural network can process. Raw text is a one-dimensional sequence of characters — just a long string. **Tokenization** converts this string into a sequence of symbols from a fixed vocabulary using Byte Pair Encoding (BPE). The algorithm starts with individual characters, then repeatedly merges the most frequent adjacent pairs into new tokens. After enough merges, you get a vocabulary — GPT-4's is roughly 100,000 tokens.

### The Transformer — A DJ's Switchboard for Meaning

The transformer architecture — billions of parameters arranged in **attention layers** — then learns a single objective: **predict the next token**. But how does it decide which tokens matter for the prediction?

Think of it like a DJ's mixing board. A DJ has dozens of channels — vocals, bass, drums, synths — and for every moment in a track, they're adjusting which channels to boost and which to suppress. The transformer's attention mechanism works the same way. For every token the model processes, it has to decide: which of the previous tokens should I pay attention to right now? When predicting the word after "The capital of France is", the attention mechanism learns to boost the signal from "France" and "capital" while suppressing "The" and "of" — just like a DJ pulling up the vocal channel and dropping the hi-hat for a breakdown.

Each attention layer is like a different DJ working the same track simultaneously — one might focus on grammatical structure, another on semantic meaning, another on positional patterns. Stack dozens of these layers and you get a system that can attend to complex, long-range relationships in text.

If you want to see this in action, Brendan Bycroft built an incredible [interactive transformer visualization](https://bbycroft.net/llm) that lets you watch tokens flow through each layer of a working model. You can see attention patterns forming in real time — which tokens are "looking at" which other tokens at each stage. It makes the abstract architecture tangible.

Given the sequence of tokens so far, the model outputs a probability distribution over the entire vocabulary — a percentage chance for each of the ~100,000 possible next tokens.

The model samples from this distribution to pick the next token, then feeds it back in and repeats. This is all it does — token in, probability distribution out, sample, repeat. Training adjusts billions of parameters so these distributions get better and better at matching what actually comes next in real text. This runs over months on thousands of GPUs, processing trillions of tokens. The model sees no labels, no instructions, no notion of right or wrong. It just learns which tokens are likely to follow which other tokens.

### Why You Get Different Answers to the Same Question

This sampling process explains something that surprises people: ask an LLM the exact same question twice and you'll often get a different answer. This isn't a bug — it's a direct consequence of how generation works.

Remember the DJ switchboard analogy. The attention layers set the levels — boosting "France" and "capital", determining that "Paris" should get 62% probability. But the DJ doesn't just slam one fader to max and mute everything else. Multiple channels have some signal. "Paris" is loudest, but "the", "a", "located" all have some probability too. The model then **rolls a weighted dice** across all ~100,000 tokens to pick the next one. Most of the time it lands on "Paris." But sometimes it lands on "the" — and now the entire sequence that follows is different, because the next prediction is conditioned on this different choice.

This is controlled by a parameter called **temperature**. At temperature 0, the model always picks the highest-probability token — deterministic, identical every time, but often robotic and repetitive. At higher temperatures, the dice is more random — more creative, more varied, but also more likely to go off the rails. Most chat interfaces default to a moderate temperature: varied enough to feel natural, constrained enough to stay relevant.

This same mechanism is why hallucinations happen. The model doesn't "know" facts and sometimes "make mistakes." There are no facts inside it — only probability distributions over tokens. If the probabilities happen to favour the correct token (which they usually do for common knowledge), the output is factually correct. If the dice lands on a less-likely-but-still-plausible token, the model confidently follows that path — generating fluent, convincing text that's wrong. It's not lying. It's not confused. It's sampling from a distribution, and sometimes the sample doesn't correspond to reality. Every word you read from an LLM is the output of a weighted dice roll — and understanding this is the single most important thing for using these tools well.

### Context Windows — How Much the Model Can See

There's a critical constraint: the model can only attend to a fixed number of tokens at a time. This is the **context window**. Think of it as the model's working memory — everything it needs to work with must fit inside this window: the system prompt, conversation history, any documents you've pasted in, and its own response so far. If the conversation grows beyond the limit, the oldest messages get silently dropped. The model doesn't remember them — they're gone.

### Why Bigger Isn't Enough — Lost in the Middle

But window size alone is misleading. A model can *accept* 200,000 tokens — but does it actually *attend* to all of them equally? Research on ["Lost in the Middle"](https://arxiv.org/abs/2307.03172) (Liu et al., 2023) found that most models don't. They pay the most attention to information at the **beginning** and **end** of their context window, while information buried in the middle gets partially ignored.

The reason is architectural. The attention mechanism computes relationships between every pair of tokens — but in practice, position matters. Tokens at the start of the context (the system prompt, the first instruction) get processed early and influence everything that follows. Tokens at the end (the most recent message, the actual question) get the most direct attention because they're closest to where the model generates its response. Tokens in the middle — page 47 of a 100-page document — are furthest from both anchors and receive the weakest attention signal.

Anthropic's Claude models have made significant progress here. On the "Needle in a Haystack" benchmark — where a specific fact is hidden at a random position within a long document and the model is asked to retrieve it — Claude achieves near-perfect recall across the entire 200K window. Many competing models score well at the beginning and end but drop significantly in the middle. Anthropic has specifically optimised for **uniform attention** across the full context, making Claude particularly strong for tasks like analysing long legal documents, reviewing entire codebases, or working through multi-chapter research papers where the critical detail could be anywhere.

This has practical implications. If you paste a 50-page document and ask a question about a detail on page 30, a model with the Lost in the Middle problem may miss it — not because it can't process 50 pages, but because attention fades in the middle. It's also one reason why [retrieval-augmented generation (RAG)](/blog/building-rag-from-scratch) works well: instead of dumping everything into the context window, you retrieve only the relevant chunks and place them where the model will attend to them most strongly.

What you get at the end is a **base model** — essentially expensive autocomplete. It's a next-token probability machine, and nothing more.

This is genuinely hard to internalise, because the base model *feels* like it understands things. It can write fluent paragraphs, produce working code, and discuss philosophy. But all of these abilities are side effects of next-token prediction on internet-scale data. The model has no concept of being helpful, truthful, or safe. It's a mirror of the internet, and it behaves accordingly.

Here's what that actually looks like. If you give a base model a prompt like:

```
User: What is the capital of France?
Assistant:
```

You might expect it to reply "Paris." But a base model doesn't know it's supposed to *answer*. It's seen millions of web pages, and this text could be from anywhere — a forum post, a quiz, a dataset, a chatbot transcript. So it might:

- Complete it correctly: `The capital of France is Paris.`
- Generate more Q&A pairs: `User: What is the capital of Germany? User: What is the capital of Spain?` — because it's pattern-matching a list of questions
- Continue with something completely unrelated — whatever text statistically follows this kind of pattern in its training data
- Produce harmful or offensive content — it has no filter, because it was never taught what's appropriate

The base model is a powerful text generator with zero alignment. It can produce anything it saw in training, weighted by probability. Getting from here to a useful assistant requires the next three stages.

GPT-2 in 2019 cost roughly $40,000 to train. Karpathy later reproduced it for about $672 with modern tooling and hardware — a striking illustration of how fast the infrastructure has improved.

## Hallucinations — When Probability Beats Truth

Before we move to the stages that fix the base model, it's worth understanding the most important consequence of next-token prediction: **hallucination**. Because the model doesn't "know" things — it predicts likely tokens — it can produce confident, fluent, completely wrong output. Some of the most famous examples reveal exactly how this works.

### The Bible Verse Problem

This one goes deeper than it first appears. Researchers at [Transluce](https://transluce.org/observability-interface) — a nonprofit AI interpretability lab co-founded by Jacob Steinhardt (UC Berkeley) and Sarah Schwettmann (MIT) — used mechanistic interpretability tools to look *inside* the model and find out exactly why it gets this wrong. What they discovered: specific neurons fire when the model encounters "9.11" and "9.8" — and those neurons have nothing to do with decimal maths. They found **"Bible verse" neurons** trained on biblical text where chapter:verse notation (Luke 9:8 comes before Luke 9:11) treats the number after the colon as a sequential integer. They found **date neurons** where 9/11 comes after 9/8. And they found **gravity neurons** where 9.8 m/s² activates unrelated physics context.

When they tested 1,280 queries comparing numbers in this format, the baseline model got just 55% right — barely better than a coin flip. When they deactivated the Bible verse, date, and gravity neurons, accuracy jumped to 77%. Roughly 0.4% of all neurons in the model relate to the Bible — a tiny fraction, but enough to corrupt decimal comparisons across the board.

### Brilliantly Smart, Bafflingly Stupid

These examples reveal something genuinely strange about LLMs: they are simultaneously the most capable and the most unreliable tools you'll ever use. The same model that can write a working compiler from a natural language description will confidently tell you that "strawberry" has two r's. The same model that can explain quantum entanglement clearly enough for a teenager to understand will insist that 9.11 is greater than 9.8. Karpathy calls this **"jagged intelligence"** — the model's capabilities aren't a smooth gradient from easy to hard. It can solve problems that stump most humans while failing tasks a child could do.

This isn't a bug that will be fixed in the next version. It's a structural consequence of how these systems work. The model doesn't *know* things — it has statistical patterns that usually correlate with truth. When the patterns align, it's astonishingly capable. When they don't, it's astonishingly wrong — and it has no way to tell the difference. It will deliver both with identical confidence.

The practical takeaway: **always cross-check**. Treat an LLM like a brilliant but unreliable colleague. Use it to draft, explore, brainstorm, and accelerate your work — but verify anything that matters. The moment you stop checking is the moment you'll get burned by a hallucination you didn't catch. These are tools that make you dramatically more productive *when you stay in the loop*. They're dangerous when you don't.

This is precisely why the remaining three training stages exist. SFT teaches models to say "I don't know" and to call tools (a calculator, a code interpreter) instead of guessing. RLHF punishes confident wrong answers. And RLVR, as we'll see, teaches the model to actually reason its way to verifiable answers.

## Stage 2: Supervised Fine-Tuning — Learning to Be Helpful

As we saw, the base model doesn't know it's supposed to be an assistant. It just predicts the next token. Supervised fine-tuning (SFT) fixes this by training the model on curated datasets of conversations — human-written examples of what a good assistant response looks like.

These datasets come from two sources. Human-curated collections like [OASST1](https://huggingface.co/datasets/OpenAssistant/oasst1) contain thousands of real conversations written by annotators. Synthetic datasets like [UltraChat](https://github.com/thunlp/UltraChat) scale this up to 1.5 million conversations generated by models and filtered for quality.

### Special Tokens — Teaching the Model Its Role

The key mechanism is **special tokens**. The raw training data for SFT isn't just plain text — it's structured with markers that tell the model who's speaking and when each turn starts and ends. These tokens don't appear in normal text. They're added to the vocabulary specifically for conversations.

```
User: What is the capital of France?
```

```
Assistant: The capital of France is Paris.
```

It's learned from thousands of examples that when text is structured as a user question, the appropriate behaviour is to provide a direct, helpful answer. SFT also teaches tool use. The training data includes examples where the assistant calls a search engine instead of guessing at facts, or runs code to verify an answer. And it teaches step-by-step reasoning — showing work instead of jumping to conclusions. None of these behaviours were present in the base model. They're all learned from examples.

The economics are dramatic. Where pretraining takes months and costs millions, SFT takes hours or days and costs orders of magnitude less. The datasets are tiny compared to pretraining data — thousands or millions of examples versus trillions of tokens. But the behavioural shift is enormous. Think of it like learning to cook by studying recipes and practising writing them out. You get good at reproducing known dishes — following established patterns reliably.

## Stage 3: RLHF — Learning from Human Preferences

SFT gets you a helpful assistant, but "helpful" is subjective. Ask a model to explain quantum mechanics and there are dozens of valid approaches — some clearer, some more accurate, some better matched to the audience. Is this summary good? Is this creative writing engaging? Is this joke funny? There's no formula for "better" — it requires human judgment.

### The Evaluation Problem

This is harder than it sounds. Consider: how do you automatically evaluate whether a joke is funny? You can't write a function for it. You can't check it against an answer key. Humour is subjective, cultural, contextual — the same joke lands differently depending on the audience. The same is true for writing quality, explanation clarity, conversational tone, and most of the things that separate a good AI assistant from a mediocre one.

You could try asking another AI to evaluate — but that just moves the problem. How do you evaluate the evaluator? This is the fundamental challenge: **evaluation of open-ended tasks is itself an open-ended task**.

### Why Ranking Works (For Humans)

Here's the clever insight behind RLHF: while *scoring* quality on an absolute scale is nearly impossible, *ranking* is something humans do naturally and reliably. Show someone two responses and ask "which is better?" — they can almost always tell you, even if they can't articulate a precise score. Consider this example:

Reinforcement Learning from Human Feedback (RLHF) uses this insight. The model generates multiple responses to the same prompt, human annotators rank them from best to worst, and a **reward model** is trained to approximate those rankings. Once trained, the reward model can score any response — scaling evaluation from a few thousand human judgments to millions of model outputs. The language model is then optimised via reinforcement learning to produce responses that score higher on the reward model.

This works well for subjective quality — tone, clarity, helpfulness, safety. The early iterations produce genuine improvements. Responses get more natural, more careful, better structured. But there's a fundamental problem: the reward model is an approximation, not ground truth. It's a neural network trained to *simulate* human preferences, and it has blind spots.

### Reward Hacking — When the Model Games the System

Given enough optimisation pressure, models learn to exploit those blind spots. This is called **reward hacking**. The model discovers patterns that score highly on the reward model but aren't actually better by human standards.

It might learn that longer responses score higher (because annotators sometimes equate length with thoroughness), so it becomes verbose. It might learn that hedging language ("It's important to note that...") scores well, so it hedges everything. It might learn to produce confident-sounding but shallow answers that pattern-match what the reward model likes. The model isn't trying to be helpful — it's trying to maximise a score, and maximising a score is not the same thing.

This is why Karpathy notes that training caps are essential — typically 100 to 1,000 RL steps. Early in RLHF training, the reward model's signal is genuinely useful and the model improves. But after too many steps, the model starts overfitting to the reward model's imperfections rather than actual quality. You have to cut it off before the gaming overtakes the gains. RLHF gets you meaningful improvements, but it has a ceiling — and pushing past that ceiling makes things worse, not better.

## Stage 4: RLVR — The 2025 Paradigm Shift

In his [2025 LLM Year in Review](https://karpathy.bearblog.dev/year-in-review-2025/), Karpathy identifies Reinforcement Learning with Verifiable Rewards (RLVR) as the single biggest paradigm change of the year. It's the breakthrough that turned LLMs from sophisticated text predictors into systems that can genuinely reason.

The key insight: for tasks with **verifiable answers** — math problems, coding challenges, logic puzzles — you don't need humans in the loop at all. The model generates an attempt, an automated verifier checks whether the answer is correct, and reinforcement learning optimises toward correct solutions. No human annotators. No imperfect reward models. Just a binary signal: right or wrong.

### DeepSeek-R1 — When the Secret Got Out

What makes RLVR remarkable isn't the mechanism — it's what emerges. Models trained with RLVR spontaneously develop reasoning strategies that nobody explicitly taught them. They start breaking problems into steps. They self-correct mid-reasoning. They explore alternative approaches when their first attempt fails. These behaviours weren't in the training data and weren't programmed — they emerged because the reward signal (correctness) incentivised whatever internal strategies produced right answers.

For years, the specifics of how RL was applied to language models were closely guarded. OpenAI's o1 model demonstrated remarkable reasoning capabilities but the company published almost nothing about how it was trained — no paper, no technical details, just the product. The techniques were widely assumed to exist but remained behind closed doors at a handful of well-funded labs.

Then, in January 2025, DeepSeek — a Chinese AI lab — published the [DeepSeek-R1 paper](https://arxiv.org/abs/2501.12948) and open-sourced the model weights. The paper laid out the full recipe: start with a base model, apply RLVR with verifiable rewards on maths and code, and watch reasoning emerge spontaneously. They showed the model developing chain-of-thought reasoning, self-verification, and backtracking — all without being explicitly trained on any of these strategies. Crucially, they published it all. The paper was a watershed moment: it demonstrated that RLVR wasn't some proprietary magic — it was a technique any sufficiently resourced team could replicate. It also showed that a relatively modest lab could produce reasoning capabilities competitive with models from labs spending orders of magnitude more. The genie was out of the bottle.

There's a direct parallel to [AlphaGo](https://www.youtube.com/watch?v=WXuK6gekU1Y). In its match against Lee Sedol, AlphaGo played Move 37 — a move no human Go player would have considered, discovered purely through self-play against a verifiable objective (winning the game). RLVR enables the same kind of emergent innovation in language. The model isn't constrained to strategies humans demonstrated — it discovers what works.

The cooking analogy makes this concrete:

> SFT is like learning to cook by reading recipes and practising writing them out. RLVR is like being given ingredients and a taste test at the end — you figure out what works through actual attempts. The first produces someone good at reproducing known recipes. The second can produce someone who discovers combinations nobody thought of, because the feedback loop rewards outcomes not process.

### Visible vs Hidden Reasoning

RLVR also created an interesting transparency divide. DeepSeek's open-source models show their full chain-of-thought reasoning — you can watch the model think step by step, backtrack, try alternative approaches, and arrive at an answer. It's like watching someone's working in the margin of an exam paper. This is invaluable for understanding *why* the model gave a particular answer and for catching errors mid-reasoning.

OpenAI's reasoning models (o1, o3) take the opposite approach: they generate reasoning traces internally but **hide them from the user**. You see a "thinking..." indicator and then get the final answer, but the actual chain of thought is redacted. The widely assumed reason is competitive: if the full reasoning traces were visible, competitors could study them to reverse-engineer OpenAI's RLVR training methodology — what reward signals they use, what reasoning patterns emerge, how the model structures its thinking. The reasoning traces are essentially a window into the proprietary training recipe. Anthropic takes a middle ground with Claude — showing a summarised version of the reasoning but not the raw trace.

This matters practically. When you can see the reasoning, you can verify it. When you can't, you're trusting a black box. It's another reason to cross-check: even when a model reasons its way to an answer, the reasoning itself can contain errors that are only visible if you can actually read it.

### Test-Time Compute — A New Scaling Axis

RLVR also unlocked a new scaling axis: **test-time compute**. Instead of making models bigger, you give them more thinking time at inference. Longer reasoning traces — more internal "steps" before producing an answer — lead to better answers. This is conceptually novel. Previously, the only way to make models smarter was to make them larger or train them on more data. Now you can make the same model smarter by letting it think longer.

Karpathy notes that compute budgets shifted significantly in 2025 — away from pretraining and toward RL. Models stayed similar sizes but got dramatically smarter through longer reinforcement learning training runs. The returns from RLVR haven't plateaued the way RLHF's did.

### Reasoning Models vs Standard Models — The New Landscape

RLVR created a split in the model landscape. There are now two distinct categories of LLM, and knowing which to use matters:

**Standard models** (sometimes called "chat" or "instruct" models) go through pretraining + SFT + RLHF. They're fast, versatile, and good at most tasks — writing, summarising, translating, answering questions, generating code. They respond in one pass without extended internal deliberation.

**Reasoning models** go through all of that *plus* RLVR. They spend extra time "thinking" before responding — generating an internal chain of reasoning that can run for thousands of tokens before producing a final answer. They're slower and more expensive to run, but dramatically better at problems that require multi-step logic.

The practical rule of thumb: use a standard model for everyday tasks (drafting emails, summarising, quick questions) and switch to a reasoning model when the problem requires genuine multi-step thinking (debugging complex code, solving maths problems, planning a system architecture). Using a reasoning model to summarise an email is like hiring a structural engineer to hang a picture frame — it'll work, but you're paying for capability you don't need.

## The Full Pipeline

The four stages form a clear progression, each building on the last:

1. **Pretraining** → broad knowledge. The model absorbs the statistical structure of human language. An internet mirror.
2. **SFT** → helpful behaviour. The model learns to follow instructions and have conversations. Recipe following.
3. **RLHF** → refined taste. Human preferences polish the model's outputs for subjective quality. The soft edges get smoothed.
4. **RLVR** → genuine reasoning. Self-discovered strategies emerge from verifiable feedback. The model learns to think.

Each stage is faster and cheaper than the one before it. Pretraining takes months and costs millions. SFT takes days. RLHF and RLVR take days to weeks. But the later stages produce disproportionate capability gains — RLVR in particular has delivered the largest qualitative improvements in model reasoning despite using a fraction of the compute budget.

## Where This Goes

Karpathy's assessment is striking: "the industry has realized less than 10% of current capability potential." The models we're using today are early. The techniques — especially RLVR and test-time compute scaling — are still being explored. The ceiling hasn't been found yet.
